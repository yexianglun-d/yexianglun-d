### under-utils

[![Java](https://img.shields.io/badge/Java-21-blue.svg)](https://github.com/yexianglun-d/under-utils)
[![Spring Boot](https://img.shields.io/badge/Spring%20Boot-3.1.x-6DB33F.svg)](https://github.com/yexianglun-d/under-utils)
[![Maven Central](https://img.shields.io/maven-central/v/io.github.yexianglun-d/under-utils-starter.svg?label=Maven%20Central)](https://central.sonatype.com/artifact/io.github.yexianglun-d/under-utils-starter)
[![Docs](https://img.shields.io/badge/docs-under--utils.howied.me-2563eb.svg)](https://under-utils.howied.me/)

[under-utils](https://github.com/yexianglun-d/under-utils) is my main Java backend infrastructure project. It is positioned as a **Java 21 / Spring Boot engineering-pattern toolkit**, not as a generic utility bucket.

It focuses on infrastructure problems that often become scattered across business services:

- request operation context propagation and safer logging boundaries;
- API rate limiting, repeat-submit protection, and service-layer idempotency;
- Redis distributed locks, cache-aside helpers, logical-expire cache rebuilds, and metrics;
- OpenAPI client governance with token refresh, signatures, idempotency headers, error decoding, retry, and sensitive-data masking;
- OpenAI-compatible AI client wrappers with streaming, tools passthrough, provider extension, and response metadata;
- MyBatis-Plus safety helpers, audit filling, pagination wrappers, and sorting allowlists;
- AES-GCM field encryption, explicit MyBatis encrypted type handling, and response masking;
- import workflows with async progress, validation errors, and error export boundaries.

The project has Maven Central publishing, a public documentation site, release notes, compatibility checks, and a module selection guide. Recent public release focus includes idempotency, security starter, MyBatis starter, safer proxy/header handling, SSE robustness, and Testcontainers compatibility.

### create-x-app

[![npm](https://img.shields.io/npm/v/create-x-app-cli.svg)](https://www.npmjs.com/package/create-x-app-cli)

[create-x-app](https://github.com/yexianglun-d/create-x-app) is a Node.js scaffolding CLI for real project startup scenarios.

It is not only a template copier. The goal is to connect the full project-creation chain:

- environment checks for Node.js, Git, pnpm, Java, and template requirements;
- interactive prompts driven by template manifests;
- built-in templates for React, Node.js, Java full-stack, admin systems, Electron, Chrome Extension, monorepo, and mobile H5;
- optional collaboration files such as `AGENTS.md`, coding rules, commitlint, and Husky;
- package-manager selection, dependency installation, Git initialization, and first commit;
- remote template fetching with local cache;
- `upgrade` command for scaffold-managed config files;
- `cxa-plugin-*` plugin discovery and installation workflow.

This project is where I explore how to make team conventions and project templates easier to reuse without turning every new repository into a manual copy-paste job.

### linux-practice

[linux-practice](https://github.com/yexianglun-d/linux-practice) is a Linux command learning platform prototype.

It combines:

- Spring Boot backend APIs and exercise seeds;
- Vue 3 + TypeScript command practice UI;
- command-attempt evaluation for spelling, arguments, and target matching;
- lab sessions, reset/check APIs, and progress tracking;
- WebSocket terminal entry for practical validation;

## Engineering Notes

I usually design projects around these questions:

- What repeated engineering problem does this solve?
- Which part should be a reusable module, and which part should stay application-specific?
- What is the expected failure behavior?
- Can the behavior be tested locally and in CI?
- Is the public API stable enough to publish?
- Does the documentation explain both usage and design boundaries?

For backend projects, I tend to avoid vague "utility method" growth. I prefer capabilities with clear runtime behavior, such as idempotency ownership, Redis TTL boundaries, cache rebuild locks, token refresh, request context propagation, masked logs, and release compatibility checks.

For CLI and tooling projects, I care about the full workflow: prompt design, environment detection, generated files, installation, rollback boundaries, upgrade strategy, and how future teams will maintain the template.

### 代表项目

#### 项目：under-utils

[under-utils](https://github.com/yexianglun-d/under-utils) 是我目前最核心的 Java 后端工程化项目。它的定位不是 Hutool、Guava 或 Apache Commons 的替代品，而是面向 Java 21 / Spring Boot 项目的工程模式工具包。

它重点解决这些在业务系统里容易写散的问题：

- 请求操作上下文传播、日志边界和敏感信息脱敏；
- 接口限流、防重复提交、服务层业务幂等；
- Redis 分布式锁、cache-aside、逻辑过期缓存重建和指标边界；
- OpenAPI 客户端治理，包括 token 刷新、签名、幂等 header、错误解码、重试和脱敏；
- OpenAI-compatible AI 客户端封装，包括同步/流式对话、tools 参数透传、provider 扩展和响应元数据；
- MyBatis-Plus 安全分页、排序白名单、审计字段填充和分页结果封装；
- AES-GCM 字段级加密、显式 MyBatis 加密 TypeHandler、响应字段脱敏；
- CSV 导入、异步导入进度、校验错误和错误导出。

这个项目已经有 Maven Central 发布、官网文档、Release Notes、模块选择指南、兼容性检查和 CI 验证。最近的公开版本重点包括幂等、安全 starter、MyBatis starter、可信代理/身份 header 边界、SSE 解析增强和 Testcontainers 兼容性。

### 我重视的工程习惯

- 先明确项目到底解决什么重复问题。
- 模块边界要清楚，不把业务一次性逻辑误做成公共工具。
- 失败语义要明确：失败后是否释放、是否可重试、是否复用第一次结果，都要说清楚。
- 能本地验证，也能在 CI 或发布前验证。
- 文档不仅写怎么用，也要说明为什么这样设计，以及不适合什么场景。
- 对外发布的能力要重视兼容性，不能随意破坏已有调用。
